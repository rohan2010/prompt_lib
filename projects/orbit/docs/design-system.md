## Visual Style

**Style:** Dark Mode First + Retro-Futuristic (hybrid)
**Source:** Section 3A "Dark Mode First" (deep bg, glowing accents, high contrast text) combined with Section 3B "Retro-Futuristic" (neon accents on dark bg, space-age grid) for the orbital visualization layer.
**Why:** The brand name is Orbit. The emotional direction demands "inevitable, intelligent, calm confidence" with a Sage + Ruler archetype. Dark Mode First provides the structured, authoritative foundation of mission control ŌĆö everything precise, high-contrast, no decoration. Retro-Futuristic provides the cosmic visual language for the orbital diagram and 7-layer system without making the rest of the site feel like a sci-fi game. The hybrid means: 90% Dark Mode First discipline, 10% Retro-Futuristic glow reserved exclusively for the interactive orbital elements.
**Quality to preserve across every section:** Engineered stillness ŌĆö every element should feel placed with gravitational precision, never decorated or arranged for visual variety.

## Emotional Brief

"This site should feel like stepping into a mission control room for the future of commerce ŌĆö quiet authority, deep intelligence, everything in its place ŌĆö not like scrolling through another SaaS landing page with blue gradients and screenshot carousels."

## Motion System

**Motion principle:** "Everything moves like orbital mechanics ŌĆö deliberate, continuous, gravitational. Elements drift into their positions as if pulled by invisible mass. Nothing bounces, nothing snaps, nothing is playful."

**Source:** Section 2A ŌĆö GSAP + ScrollTrigger + Lenis for scroll choreography. Section 2C ŌĆö Three.js for WebGL orbital diagram.

| Token | Value | Use |
|---|---|---|
| --ease-out | cubic-bezier(0.16, 1, 0.3, 1) | Most scroll entrances |
| --ease-gravitational | cubic-bezier(0.25, 0.46, 0.45, 0.94) | Orbital motion, drift entrances |
| --ease-in-out | cubic-bezier(0.4, 0, 0.2, 1) | Section transitions |
| --ease-decel | cubic-bezier(0.0, 0.0, 0.2, 1) | Elements settling into position |
| --duration-micro | 200ms | Hover states, button feedback |
| --duration-enter | 500ms | Scroll entrance animations |
| --duration-slow | 1000ms | Hero reveals, section wipes |
| --duration-orbital | 60000ms | Full orbital rotation cycle (1 minute) |

**GSAP ScrollTrigger defaults:**
```javascript
{
  start: "top 85%",
  end: "top 20%",
  toggleActions: "play none none none"
}
```

**Lenis configuration:**
```javascript
{
  duration: 1.2,
  easing: (t) => Math.min(1, 1.001 - Math.pow(2, -10 * t)),
  orientation: 'vertical',
  smoothWheel: true,
  smoothTouch: false,
  touchMultiplier: 2
}
```

**Entrance animation patterns:**
- **Drift up:** `y: 40, opacity: 0` -> `y: 0, opacity: 1` over `--duration-enter` with `--ease-out`
- **Drift in from side:** `x: 60, opacity: 0` -> `x: 0, opacity: 1` ŌĆö used sparingly for comparison sections
- **Fade only:** `opacity: 0` -> `opacity: 1` ŌĆö for elements that should not move, only appear
- **Scale reveal:** `scale: 0.95, opacity: 0` -> `scale: 1, opacity: 1` ŌĆö for cards and contained elements
- **Stagger:** 0.08s between siblings in a group (cards, list items, layer nodes)

**Rules:**
- No bounce easing anywhere on the site
- No spring physics ŌĆö this is gravitational, not elastic
- Never animate two consecutive sections with the same entrance pattern
- The orbital diagram has perpetual motion independent of scroll ŌĆö it is always alive
- All GSAP timelines must register with ScrollTrigger for proper cleanup
- `prefers-reduced-motion: reduce` ŌĆö disable all GSAP animations, Three.js orbital rotation, parallax. Show all content statically. The orbital diagram renders as a static SVG fallback with all 7 nodes visible.

## Orbital Diagram ŌĆö Technical Specification

This is the most complex component on the site. It appears in three contexts:
1. **Hero background** (Section 2) ŌĆö full-bleed, ambient, no interaction
2. **Interactive centerpiece** (Section 5) ŌĆö full interaction, detail expansion
3. **CTA background** (Section 8) ŌĆö faded callback, no interaction

### Rendering Approach
Use **HTML Canvas 2D** (not Three.js) for the primary implementation. Rationale:
- The diagram is fundamentally 2D ŌĆö elliptical orbits viewed from above
- Canvas 2D is lighter than WebGL, better performance on mobile
- GSAP can drive canvas animation via requestAnimationFrame
- Three.js should only be introduced if a 3D perspective tilt effect is required (stretch goal)

If the builder determines 3D perspective is necessary, use **React Three Fiber** (Section 9 of resource library) with **Drei** helpers for camera control.

### Diagram Structure
```
Canvas dimensions: 100% of container width, aspect-ratio 1:1 (square)
Center point: (cx, cy) = center of canvas

Central node: "MERCHANT" ŌĆö positioned at (cx, cy)
  - Radius: 28px
  - Fill: --color-text at 10% opacity
  - Border: 1.5px --color-text at 40% opacity
  - Label: "MERCHANT" in --font-mono, --color-text

Orbit rings: 3 concentric ellipses
```

### Mobile Fallback
Below 768px viewport width:
- Replace canvas orbital diagram with a static or lightly-animated SVG
- Or: replace with a scroll-driven sequential reveal ŌĆö each layer gets a horizontal card that slides in as user scrolls, with the layer code, color dot, and brief description
- The key requirement is that the 7-layer concept is communicated clearly even without the animated orbit

### Performance
- Canvas renders at device pixel ratio capped at 2 (no need for 3x on high-DPI)
- Use `requestAnimationFrame` with frame skipping on low-end devices (target 30fps minimum)
- Intersection Observer: only run the animation loop when the canvas is in viewport
- Offscreen canvases for the hero and CTA variants can share a single animation state
